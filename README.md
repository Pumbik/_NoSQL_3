# MovieLens 1M: Граф знань та Рекомендаційна система на базі Neo4j

## Про проєкт
Цей проєкт присвячений дослідженню графових баз даних як потужного фундаменту для побудови сучасних рекомендаційних систем. Перехід від традиційних реляційних структур до графових моделей дозволяє елегантно та ефективно вирішувати задачі колаборативної фільтрації, пошуку шляхів та видобування прихованих патернів. 

Робота охоплює повний цикл обробки даних: від проєктування графової схеми та імпорту реального датасету (MovieLens 1M) до виконання складних аналітичних запитів мовою Cypher та застосування алгоритмів Graph Data Science (PageRank, Louvain, Dijkstra) для глибокого аналізу зв'язків між користувачами та контентом.

## Мета та завдання
* Спроєктувати оптимізовану графову схему для ефективного проходження по ребрах (graph traversal).
* Реалізувати ETL-процес: конвертацію `.dat` файлів у `.csv` та батчеве завантаження даних у Neo4j.
* Опанувати мову Cypher для фільтрації, генерації рекомендацій та пошуку супервузлів.
* Застосувати алгоритми GDS для кластеризації та аналізу топології графа.
* Провести порівняльний аналіз графового та реляційного підходів.

## Частина 1: Проєктування схеми

Для побудови рекомендаційної системи на базі датасету MovieLens 1M була обрана наступна структура графа:

### Опис сутностей

**Вузли (Nodes):**
1. **`User`** — представляє користувача.
   * *Властивості:* `userId` (ідентифікатор), `gender` (стать), `age` (вікова група), `occupation` (код професії), `zip` (поштовий індекс).
2. **`Movie`** — представляє фільм.
   * *Властивості:* `movieId` (ідентифікатор), `title` (назва), `year` (рік випуску).
3. **`Genre`** — представляє жанр кінематографа.
   * *Властивості:* `name` (назва жанру, наприклад "Action" або "Comedy").

**Ребра (Relationships):**
1. **`[:RATED]`** — дія оцінювання фільму користувачем. 
   * *Напрямок:* `(User) -> (Movie)`
   * *Властивості:* `rating` (оцінка від 1 до 5), `timestamp` (час оцінювання).
2. **`[:HAS_GENRE]`** — належність фільму до певного жанру.
   * *Напрямок:* `(Movie) -> (Genre)`
   * *Властивості:* немає.

### Структура графа (ASCII-діаграма)

```text
  +-----------------------+
  |        :User          |
  |-----------------------|
  | userId: String/Int    |
  | gender: String        |
  | age: Integer          |
  | occupation: Integer   |
  | zip: String           |
  +-----------------------+
              |
              | :RATED
              | { rating: Integer, 
              |   timestamp: Integer }
              v
  +-----------------------+        :HAS_GENRE         +-------------------+
  |       :Movie          |-------------------------->|      :Genre       |
  |-----------------------|                           |-------------------|
  | movieId: String/Int   |                           | name: String      |
  | title: String         |                           +-------------------+
  | year: Integer         |
  +-----------------------+

```

### Обґрунтування архітектурних рішень
1. Які сутності стали вузлами, а які — ребрами? Чому?
    * Вузлами стали User, Movie та Genre. Це самостійні, незалежні сутності предметної області, які містять власні атрибути і є основними об'єктами пошуку. Ребрами стали зв'язки RATED та HAS_GENRE. У графових базах даних ребра найкраще підходять для відображення дій, взаємодій або структурної підпорядкованості між вузлами. Користувач зробив дію (оцінив фільм), а фільм структурно належить до жанру.
2. Оцінка користувача за фільм — це ребро (User)-[:RATED]->(Movie) чи окремий вузол (Rating)? Аргументуйте своє рішення. Це не риторичне запитання: в обох підходів є реальні trade-off-и.
    * У цій схемі оцінка реалізована як ребро [:RATED] з властивостями. Графові бази оптимізовані для проходження по ребрах. Розміщення оцінки на ребрі дозволяє шукати фільми одним "стрибком": (u:User)-[r:RATED]->(m:Movie). Якби Rating був окремим вузлом, це створило б понад 1 мільйон зайвих вузлів і 2 мільйони зайвих ребер, що суттєво уповільнило б виконання запитів.
3. Чому жанри фільму вигідніше зберігати як окремі вузли (Genre), а не як список у властивості вузла Movie?
    * Зберігання жанрів як окремих вузлів робить граф потужнішим інструментом для колаборативної фільтрації. 
    Це забезпечує:
        - Швидкість пошуку: Щоб знайти всі комедії, базі достатньо знайти один вузол Genre і пройтися по вхідних ребрах <-[:HAS_GENRE]-, уникаючи повного сканування всіх фільмів.
        - Зручність рекомендацій: Вузли жанрів дозволяють легко знаходити спільні інтереси між користувачами через графові патерни (наприклад, користувачі, які оцінили фільми однакових жанрів).


## Частина 2: Завантаження даних

Для завантаження конвертованих CSV-файлів у Neo4j був створений скрипт `part2_load.cypher`. Всі етапи виконувались у чіткій послідовності: створення вузлів -> індексування -> створення зв'язків.

### 1. Завантаження вузлів
* **Користувачі (`User`):** Я використовую `MERGE` за `userId`, що гарантує відсутність дублікатів (ідемпотентність запиту). Властивості записуються в блоці `ON CREATE SET`, причому цифрові значення обов'язково конвертуються через `toInteger()`, щоб уникнути зберігання чисел як тексту.
* **Фільми (`Movie`):** Оскільки оригінальний датасет містить рік у самій назві (наприклад, *"Toy Story (1995)"*), я використал строкові функції Cypher (`substring` та `trim`), щоб розбити цей рядок на дві окремі властивості: чисту назву `title` та числовий рік `year`. Це дозволить зручно фільтрувати фільми за роками.
* **Жанри (`Genre`):** Жанри збережені в CSV як один рядок з роздільником `|` (наприклад, *Animation|Children's|Comedy*). За допомогою команди `UNWIND split(row.genres, '|')` я розбиваю цей рядок на масив і для кожного елемента створюю окремий унікальний вузол `Genre` за допомогою `MERGE`.

### 2. Індекси
Перед тим як створювати зв'язки, необхідно створити B-Tree індекси для `userId`, `movieId` та `name` (для жанрів). 
**Чому це важливо:** При створенні зв'язків (наприклад, мільйона ребер `RATED`) Neo4j повинен спочатку знайти відповідного користувача і фільм (`MATCH`). Без індексів база виконувала б повне сканування (Full Node Scan) для кожного з мільйона рядків, що зайняло б години. З індексами цей пошук відбувається за долі секунди (`O(log N)`).

### 3. Завантаження ребер (Зв'язків)
* **Зв'язки `[:HAS_GENRE]`:** Завантажуються звичайним `LOAD CSV`, оскільки їх відносно небагато. Я знаходжу фільм, розбиваю його жанри через `UNWIND`, знаходжу відповідні вузли жанрів і з'єдную їх `MERGE (m)-[:HAS_GENRE]->(g)`.
* **Зв'язки `[:RATED]` (1 000 209 записів):** Такий обсяг не можна завантажити однією транзакцією — Neo4j вичерпає оперативну пам'ять (Heap Space) або видасть Timeout. Тому я використал бібліотеку APOC і процедуру `apoc.periodic.iterate`. 
    * Вона бере перший запит (читання CSV) і передає його результати в другий запит (створення зв'язків) пачками (батчами).
    * `batchSize: 10000` — означає, що транзакція фіксується кожні 10 тисяч записів, очищуючи пам'ять.
    * `parallel: false` — **це захист від блокувань (Deadlocks).** При паралельному завантаженні кілька потоків можуть одночасно спробувати оновити або заблокувати один і той самий вузол фільму чи користувача, що призведе до краху транзакції. Оскільки я використовую `MERGE`, послідовне завантаження (`false`) є найбільш безпечним та стабільним підходом.

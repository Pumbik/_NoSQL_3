# MovieLens 1M: Граф знань та Рекомендаційна система на базі Neo4j

## Про проєкт

Цей проєкт присвячений дослідженню графових баз даних як потужного фундаменту для побудови сучасних рекомендаційних систем. Перехід від традиційних реляційних структур до графових моделей дозволяє елегантно та ефективно вирішувати задачі колаборативної фільтрації, пошуку шляхів та видобування прихованих патернів.

Робота охоплює повний цикл обробки даних: від проєктування графової схеми та імпорту реального датасету (MovieLens 1M) до виконання складних аналітичних запитів мовою Cypher та застосування алгоритмів Graph Data Science (PageRank, Louvain, Dijkstra) для глибокого аналізу зв'язків між користувачами та контентом.

## Мета та завдання

- Спроєктувати оптимізовану графову схему для ефективного проходження по ребрах (graph traversal).
- Реалізувати ETL-процес: конвертацію `.dat` файлів у `.csv` та батчеве завантаження даних у Neo4j.
- Опанувати мову Cypher для фільтрації, генерації рекомендацій та пошуку супервузлів.
- Застосувати алгоритми GDS для кластеризації та аналізу топології графа.
- Провести порівняльний аналіз графового та реляційного підходів.

## Частина 1: Проєктування схеми

Для побудови рекомендаційної системи на базі датасету MovieLens 1M була обрана наступна структура графа:

### Опис сутностей

**Вузли (Nodes):**

1. **`User`** — представляє користувача.
   - _Властивості:_ `userId` (ідентифікатор), `gender` (стать), `age` (вікова група), `occupation` (код професії), `zip` (поштовий індекс).
2. **`Movie`** — представляє фільм.
   - _Властивості:_ `movieId` (ідентифікатор), `title` (назва), `year` (рік випуску).
3. **`Genre`** — представляє жанр кінематографа.
   - _Властивості:_ `name` (назва жанру, наприклад "Action" або "Comedy").

**Ребра (Relationships):**

1. **`[:RATED]`** — дія оцінювання фільму користувачем.
   - _Напрямок:_ `(User) -> (Movie)`
   - _Властивості:_ `rating` (оцінка від 1 до 5), `timestamp` (час оцінювання).
2. **`[:HAS_GENRE]`** — належність фільму до певного жанру.
   - _Напрямок:_ `(Movie) -> (Genre)`
   - _Властивості:_ немає.

### Структура графа (ASCII-діаграма)

```text
  +-----------------------+
  |        :User          |
  |-----------------------|
  | userId: String/Int    |
  | gender: String        |
  | age: Integer          |
  | occupation: Integer   |
  | zip: String           |
  +-----------------------+
              |
              | :RATED
              | { rating: Integer,
              |   timestamp: Integer }
              v
  +-----------------------+        :HAS_GENRE         +-------------------+
  |       :Movie          |-------------------------->|      :Genre       |
  |-----------------------|                           |-------------------|
  | movieId: String/Int   |                           | name: String      |
  | title: String         |                           +-------------------+
  | year: Integer         |
  +-----------------------+

```

### Обґрунтування архітектурних рішень

1. Які сутності стали вузлами, а які — ребрами? Чому?
   - Вузлами стали User, Movie та Genre. Це самостійні, незалежні сутності предметної області, які містять власні атрибути і є основними об'єктами пошуку. Ребрами стали зв'язки RATED та HAS_GENRE. У графових базах даних ребра найкраще підходять для відображення дій, взаємодій або структурної підпорядкованості між вузлами. Користувач зробив дію (оцінив фільм), а фільм структурно належить до жанру.
2. Оцінка користувача за фільм — це ребро (User)-[:RATED]->(Movie) чи окремий вузол (Rating)? Аргументуйте своє рішення. Це не риторичне запитання: в обох підходів є реальні trade-off-и.
   - У цій схемі оцінка реалізована як ребро [:RATED] з властивостями. Графові бази оптимізовані для проходження по ребрах. Розміщення оцінки на ребрі дозволяє шукати фільми одним "стрибком": (u:User)-[r:RATED]->(m:Movie). Якби Rating був окремим вузлом, це створило б понад 1 мільйон зайвих вузлів і 2 мільйони зайвих ребер, що суттєво уповільнило б виконання запитів.
3. Чому жанри фільму вигідніше зберігати як окремі вузли (Genre), а не як список у властивості вузла Movie?
   - Зберігання жанрів як окремих вузлів робить граф потужнішим інструментом для колаборативної фільтрації.
     Це забезпечує: - Швидкість пошуку: Щоб знайти всі комедії, базі достатньо знайти один вузол Genre і пройтися по вхідних ребрах <-[:HAS_GENRE]-, уникаючи повного сканування всіх фільмів. - Зручність рекомендацій: Вузли жанрів дозволяють легко знаходити спільні інтереси між користувачами через графові патерни (наприклад, користувачі, які оцінили фільми однакових жанрів).

## Частина 2: Завантаження даних

Для завантаження конвертованих CSV-файлів у Neo4j був створений скрипт `part2_load.cypher`. Всі етапи виконувались у чіткій послідовності: створення вузлів -> індексування -> створення зв'язків.

### 1. Завантаження вузлів

- **Користувачі (`User`):** Я використовую `MERGE` за `userId`, що гарантує відсутність дублікатів (ідемпотентність запиту). Властивості записуються в блоці `ON CREATE SET`, причому цифрові значення обов'язково конвертуються через `toInteger()`, щоб уникнути зберігання чисел як тексту.
- **Фільми (`Movie`):** Оскільки оригінальний датасет містить рік у самій назві (наприклад, _"Toy Story (1995)"_), я використал строкові функції Cypher (`substring` та `trim`), щоб розбити цей рядок на дві окремі властивості: чисту назву `title` та числовий рік `year`. Це дозволить зручно фільтрувати фільми за роками.
- **Жанри (`Genre`):** Жанри збережені в CSV як один рядок з роздільником `|` (наприклад, _Animation|Children's|Comedy_). За допомогою команди `UNWIND split(row.genres, '|')` я розбиваю цей рядок на масив і для кожного елемента створюю окремий унікальний вузол `Genre` за допомогою `MERGE`.

### 2. Індекси

Перед тим як створювати зв'язки, необхідно створити B-Tree індекси для `userId`, `movieId` та `name` (для жанрів).
**Чому це важливо:** При створенні зв'язків (наприклад, мільйона ребер `RATED`) Neo4j повинен спочатку знайти відповідного користувача і фільм (`MATCH`). Без індексів база виконувала б повне сканування (Full Node Scan) для кожного з мільйона рядків, що зайняло б години. З індексами цей пошук відбувається за долі секунди (`O(log N)`).

### 3. Завантаження ребер (Зв'язків)

- **Зв'язки `[:HAS_GENRE]`:** Завантажуються звичайним `LOAD CSV`, оскільки їх відносно небагато. Я знаходжу фільм, розбиваю його жанри через `UNWIND`, знаходжу відповідні вузли жанрів і з'єдную їх `MERGE (m)-[:HAS_GENRE]->(g)`.
- **Зв'язки `[:RATED]` (1 000 209 записів):** Такий обсяг не можна завантажити однією транзакцією — Neo4j вичерпає оперативну пам'ять (Heap Space) або видасть Timeout. Тому я використал бібліотеку APOC і процедуру `apoc.periodic.iterate`.
  - Вона бере перший запит (читання CSV) і передає його результати в другий запит (створення зв'язків) пачками (батчами).
  - `batchSize: 10000` — означає, що транзакція фіксується кожні 10 тисяч записів, очищуючи пам'ять.
  - `parallel: false` — **це захист від блокувань (Deadlocks).** При паралельному завантаженні кілька потоків можуть одночасно спробувати оновити або заблокувати один і той самий вузол фільму чи користувача, що призведе до краху транзакції. Оскільки я використовую `MERGE`, послідовне завантаження (`false`) є найбільш безпечним та стабільним підходом.

## Частина 3: Запити різної складності (Cypher)

Усі запити збережено у файлі `queries/part3.cypher`. Нижче наведено пояснення логіки кожного з них:

### Базові запити

**Запит 1: Фільми жанру «Thriller» із рейтингом > 4.0**
Цей запит використовує два послідовні `MATCH`. Перший швидко фільтрує вузли фільмів, залишаючи лише ті, що з'єднані з жанром "Thriller". Другий знаходить всі оцінки до цих фільмів. Ключовим тут є агрегаційна функція `avg(r.rating)` та конструкція `WITH`, яка групує результати за фільмом (`m`) перед тим, як застосувати фільтр `WHERE avgRating > 4.0`. Також додано умову `totalRatings > 10`, щоб відсіяти фільми з однією випадковою оцінкою "5".

**Запит 2: Користувачі, які поставили > 50 оцінок "5"**
Запит проходить по всіх ребрах `RATED` і одразу відсіює ті, де оцінка не дорівнює 5. Далі він групує дані за користувачем (`WITH u`) і підраховує кількість таких фільмів за допомогою `count(m)`. Оператор `WHERE` після агрегації діє як `HAVING` у SQL, фільтруючи користувачів.

### Запити середнього рівня

**Запит 3: Спільні високооцінені фільми для двох користувачів**
Це класичний приклад того, де граф знищує реляційну БД. Ми описуємо патерн як один ланцюжок: `(u1)-[r1]->(m)<-[r2]-(u2)`. Neo4j знайде всі фільми, куди "сходяться" ребра від обох користувачів одночасно. У SQL для цього знадобився б складний `INNER JOIN` таблиці оцінок самої з собою.

**Запит 4: Жанри зі стабільно високими оцінками**
Запит проходить повний шлях від користувача до жанру: `(Genre)<-[:HAS_GENRE]-(Movie)<-[:RATED]-(User)`. Потім він групує всі оцінки за назвою жанру та вираховує середнє значення. Додано фільтр кількості оцінок, щоб забезпечити статистичну значущість.

### Складні запити

**Запит 5: Колаборативна фільтрація ("Користувачі зі схожими смаками також дивилися")**
Запит розділений на дві частини за допомогою `WITH`.

1. Спочатку ми знаходимо "схожих користувачів" (`similar`): тих, хто високо оцінив ті ж фільми, що і цільовий користувач. Чим більше спільних фільмів, тим вищий `similarityScore`.
2. Потім ми дивимося, які ще фільми ці схожі користувачі високо оцінили.
3. Критичний момент — конструкція `NOT EXISTS { MATCH (target)-[:RATED]->(recMovie) }`, яка фільтрує рекомендації, виключаючи ті фільми, які цільовий користувач вже бачив. Зрештою фільми ранжуються за сумою оцінок схожості.

**Запит 6: Найкоротший шлях (Shortest Path)**
Використовує вбудовану графову функцію `shortestPath()`. Вона шукає найкоротшу кількість "стрибків" (hops) будь-якого напрямку між двома вузлами (ребра `RATED` ігнорують напрямок у цьому контексті). Оператор `*..6` обмежує пошук глибиною 6 кроків, щоб уникнути перевантаження бази.

---

### Відповіді на теоретичні питання (Інтерпретація шляху)

**Що означає довжина шляху в даному контексті?**
Довжина шляху в графі визначається кількістю пройдених ребер (кроків/hops) від початкового до кінцевого вузла. Оскільки граф двочастковий щодо оцінок (ребра йдуть лише від Користувача до Фільму), будь-який шлях між двома користувачами складатиметься виключно з парної кількості ребер.

- **Довжина 2 (1 хоп туди, 1 назад):** `(Користувач A) -[:RATED]-> (Фільм 1) <-[:RATED]- (Користувач B)`. Це означає прямий зв'язок: два користувачі подивилися та оцінили один і той самий фільм. Вони мають спільний інтерес.
- **Довжина 4:** `(Користувач A) -> (Фільм 1) <- (Користувач B) -> (Фільм 2) <- (Користувач C)`. Це зв'язок через посередника ("друг мого друга"). Користувач A і C не дивилися жодного спільного фільму, але їх об'єднує Користувач B, який має спільний фільм з A і спільний фільм з C.
- **Довжина 6:** `(Користувач A) -> (Фільм 1) <- (Користувач B) -> (Фільм 2) <- (Користувач C) -> (Фільм 3) <- (Користувач D)`. Це вже дуже далекий зв'язок ("теорія шести рукостискань"). На практиці в щільному датасеті (як MovieLens) майже будь-які два користувачі будуть пов'язані шляхом довжини 4 або 6, оскільки існують дуже популярні фільми (блокбастери), які дивилися майже всі.

## Частина 4: Виявлення супервузлів

Використав функцію `COUNT { (n)--() }`, яка дозволяє отримати ступінь вузла (degree) за `O(1)`, оскільки Neo4j кешує кількість ребер у метаданих вузла.

**Відповіді на теоретичні запитання:**

### 1. Які вузли виявилися супервузлами? Скільки у них зв’язків?

Після виконання запитів виходить, що супервузлами в цьому датасеті є:

1. **Фільми-блокбастери:** Наприклад, _"American Beauty (1999)"_ або _"Star Wars: Episode IV..."_ мають понад 3000 зв'язків (оцінок).
2. **Вузли жанрів:** Оскільки жанрів лише 18, а фільмів майже 3800, такі вузли як _"Comedy"_ або _"Drama"_ мають величезну кількість вхідних ребер (наприклад, Drama може мати понад 1000 зв'язків `HAS_GENRE`).
3. **Гіперактивні користувачі:** Деякі вузли `User` мають понад 1000-2000 вихідних зв'язків `RATED`.

### 2. Чому запит, що зачіпає такий вузол, працює повільніше, ніж запит по «звичайному» вузлу з тими самими індексами?

Індекси (наприклад, на `movieId` або `name`) допомагають Neo4j миттєво знайти **сам вузол** (стартову точку). Але індекси не індексують **ребра**, що відходять від цього вузла.
Коли рушій бази даних стає на супервузол (наприклад, фільм "Star Wars") і йому потрібно знайти конкретного користувача, який його оцінив, базі доводиться завантажувати в оперативну пам'ять і перебирати **усі 3000+ ребер** цього вузла одне за одним. Якщо ж запит проходить через супервузол транзитом (наприклад, `(User)->(Movie)->(User)`), виникає проблема "комбінаторного вибуху" (Combinatorial Explosion), і кількість шляхів для перевірки зростає в геометричній прогресії, що "вішає" базу.

### 3. Яку конкретну стратегію з лекцій ви б застосували для цього датасету? (Що робити з жанрами?)

Жанрові вузли в цьому датасеті є класичними супервузлами. У Частині 1 ми виділили їх як окремі вузли, але тепер бачимо trade-off: будь-яка колаборативна фільтрація через жанр (наприклад, `(m1)-[:HAS_GENRE]->(g:Genre)<-[:HAS_GENRE]-(m2)`) миттєво потягне за собою тисячі фільмів, уповільнюючи пошук.

**Стратегії вирішення:**

1. **Згортання у властивості (Refactoring):** Найкращий підхід для вузлів, яких дуже мало (всього 18 жанрів) — це знищити вузли `(Genre)` і перенести їх назад як масив властивостей до фільму: `Movie {genres: ['Action', 'Comedy']}`. Тоді фільтрація відбуватиметься через індекси на масивах, а не через граф, усуваючи супервузол.
2. **Специфічні типи ребер:** Замість універсального ребра `HAS_GENRE` та вузла `Genre`, можна створити конкретні типи ребер: `(Movie)-[:IS_COMEDY]->()`. Це розбиває щільність одного ребра на кілька різних, що прискорює проходження (traversal).
3. **Обмеження обходу (Query Tuning):** Якщо ми залишаємо жанри як вузли, під час написання запитів ми маємо штучно обмежувати кількість результатів (`LIMIT`) або фільтрувати ребра за часом (`timestamp`), щоб алгоритм не намагався прочитати всі тисячі зв'язків.

## Частина 5: Графові алгоритми через GDS

### 5.1. PageRank на графі фільмів

Для запуску алгоритму я спочатку матеріалізували зв'язки `CO_RATED` (спільні високі оцінки) між фільмами, після чого створив in-memory проєкцію графа `movieGraph`. Запит `gds.pageRank.stream` обчислив транзитивну важливість кожного фільму, враховуючи вагу цих зв'язків.

**Відповідь на теоретичне запитання:**

- **Що означає високий PageRank для фільму в цьому графі? Це просто “популярний фільм” чи щось інше?**

### 5.2. Виявлення спільнот (Louvain)

У цій частині я побудувал граф схожості користувачів (ребро `SIMILAR` створювалося, якщо двоє користувачів високо оцінили однакові фільми). Після створення in-memory проєкції `userSimilarity`, я застосувал алгоритм Louvain.

- **Крок 3 (`gds.louvain.write`):** Виконує алгоритм і записує ідентифікатор знайденої спільноти (`communityId`) як нову властивість безпосередньо у вузол `User`. Це дозволяє далі працювати з кластерами через звичайний Cypher. Наступний запит групує користувачів за `communityId` і виводить топ-10 найбільших груп.

## Частина 6: Аналіз і висновки

### 1. Граф vs SQL: Сила Cypher у зв'язках
Графова модель беззаперечно перемагає реляційну (SQL) там, де головною цінністю є *зв'язки* між даними, а не самі дані. Найяскравіші приклади з Частини 3 — це Запит 5 (Колаборативна фільтрація) та Запит 6 (Найкоротший шлях).

У **Запиті 5** я шукав фільми, які сподобалися людям зі схожими смаками. У Cypher це описується інтуїтивним візуальним патерном: `(target)-[:RATED]->(m)<-[:RATED]-(similar)-[:RATED]->(recMovie)`. 
Щоб написати еквівалент у SQL, довелося б зробити щонайменше три `JOIN` величезної таблиці `Ratings` (на мільйон рядків) самої із собою, згрупувати результати, відсортувати та використати `NOT IN` або `LEFT JOIN ... IS NULL` для відсіювання вже переглянутих фільмів. 

**Приклад еквівалентного SQL для Запиту 3 (Спільні фільми для двох користувачів):**
```sql
-- У SQL це вимагає само-об'єднання (Self-Join) таблиці оцінок
SELECT m.title, r1.rating AS ratingUser1, r2.rating AS ratingUser2
FROM Movies m
JOIN Ratings r1 ON m.movieId = r1.movieId
JOIN Ratings r2 ON m.movieId = r2.movieId
WHERE r1.userId = 1 AND r1.rating >= 4
  AND r2.userId = 2 AND r2.rating >= 4;
```
А для **Запиту 6 (Найкоротший шлях)** стандартного SQL взагалі недостатньо. Реляційні бази не вміють шукати шлях невідомої довжини. Для цього довелося б писати надскладні рекурсивні запити (Recursive CTE), які працюють вкрай повільно, оскільки DataEngine не оптимізований для проходження по графах.

### 2. Де граф програє: Слабкі місця Neo4j
Графові бази даних не є "срібною кулею". Вони жахливо справляються із задачами **глобальної агрегації, повнотекстового пошуку та побудови класичної звітності**. 

Реляційна модель (особливо колонкові бази даних) підійшла б значно краще для таких задач із цим датасетом:
* **Глобальні звіти:** Наприклад, *"Вивести середній рейтинг усіх фільмів, згрупований за поштовим індексом (zip) та віковою групою користувачів за всі роки"*. SQL просто просканує таблицю по стовпцях, підсумує дані і видасть результат майже міттево. Neo4j для цього доведеться фізично "пройтися" по кожному з 1 000 000 ребер `RATED`, що викличе величезне навантаження на процесор і пам'ять.
* **Масовий експорт (ETL):** Якщо потрібно вивантажити всю базу в єдиний "плоский" файл (CSV) для навчання машинного навчання, SQL зробить це однією командою `SELECT *`. У Neo4j вивантаження графа в табличний вигляд вимагає розгортання зв'язків і є повільним процесом.

### 3. Покращення схеми: Оптимізація під конкретні запити
Схему з Частини 1 варто було б рефакторити для прискорення роботи:

* **Оптимізація для Запиту 1 (та проблеми супервузлів з Частини 4):**
  У першому запиті я шукав фільми жанру "Thriller". Зараз база змушена йти до супервузла `(Genre {name: 'Thriller'})` і перебирати тисячі вхідних ребер. 
  *Покращення:* Знищити вузли `Genre` і перенести їх у масив властивостей самого фільму: `(m:Movie {genres: ['Action', 'Thriller']})`. Якщо створити індекс на цю властивість, фільтрація за жанром відбуватиметься миттєво без проходження графом.
* **Оптимізація для Запитів 2 та 5 (Швидка фільтрація оцінок):**
  У цих запитах я постійно використовував `WHERE r.rating >= 4` або `r.rating = 5`. Neo4j змушений відкривати властивості кожного ребра, щоб перевірити цифру.
  *Покращення:* Винести оцінку в саму **назву типу ребра** (Relationship Type). Замість одного ребра `[:RATED {rating: 5}]`, створювати специфічні ребра: `[:RATED_5]`, `[:RATED_4]`, `[:RATED_3]` і т.д. Тоді запит `MATCH (u)-[:RATED_5]->(m)` відпрацює блискавично, оскільки рушій бази даних фільтрує типи ребер на апаратному рівні ще до того, як читатиме їхні властивості.
